Ответы на вопросы

1. Какая команда оказалась самой непонятной до практики?

Наиболее сложной для понимания оказалась команда git checkout и в целом работа с ветками. Ранее я предполагал, что главная ветка main создаётся автоматически сразу после инициализации репозитория. Однако на практике стало ясно, что до появления первого коммита ветка фактически не существует. Также возникали трудности при переключении между ветками и понимании того, в какой ветке я нахожусь в данный момент. После выполнения практики стало понятнее, как создаются ветки, как происходит переключение и зачем это необходимо при параллельной разработке.

2. Почему важно настраивать user.name и user.email перед коммитами?

Настройка имени пользователя и электронной почты необходима для корректной идентификации автора изменений. Если эти параметры не заданы, Git не сможет правильно подписывать коммиты. Это приводит к путанице в истории проекта и затрудняет совместную работу, особенно если в разработке участвуют несколько человек. Корректно настроенные данные позволяют отслеживать вклад каждого разработчика и обеспечивают прозрачность истории изменений.

3. В каком случае git revert безопаснее, чем git reset?

Команда git revert считается более безопасной при работе с общим (удалённым) репозиторием. Она создаёт новый коммит, который отменяет изменения предыдущего, не изменяя историю проекта. Это важно, если другие разработчики уже получили эти изменения.

В отличие от неё, git reset изменяет историю коммитов, что может привести к конфликтам и проблемам синхронизации у других участников команды. Поэтому при коллективной разработке предпочтительнее использовать git revert, чтобы избежать потери данных и нарушения истории проекта.

4. Как .gitignore мог бы спасти проект от утечки данных?

Файл .gitignore позволяет исключить из отслеживания определённые файлы и каталоги. Это особенно важно для конфиденциальной информации, например файлов с паролями, ключами доступа, настройками окружения (.env), логами или временными файлами.

Если такие файлы случайно добавить в репозиторий, они станут доступны всем участникам проекта или даже публично в случае открытого репозитория. Правильно настроенный .gitignore помогает предотвратить подобные ситуации и повышает безопасность проекта.
